Hadoop

2019-07-18

Introduction

The Apache Hadoop software library is a framework that allows for the distributed processing of large data sets across clusters of computers using simple programming models.

Hadoop 2.x takes Hadoop beyond batch application: Allow for multiple ways to interact with data stored in HDFS: (1) batch with MapReduce(MR), (2) streaming with Spark/Storm, (3) interactive SQL with Spark SQL, Hive.

Apache Hadoop contains the following modules:

  1. Hadoop Common- contains libraries and utilities needed by other Hadoop modules.
  2. Hadoop Distributed File System(HDFS): A distributed system for storing data
  3. Hadoop MapReduce: A programming model for large scale data processing
  4. Hadoop YARN: resource-management and task scheduling.

Hadoop 1.x: Key Components

  1. Job Tracker@master node

    Keeps track of all MapReduce jobs that are running on various nodes.

  2. Name Node@master node

    Store metadata: file/chuck namespaces, file-to chunk mapping, location of each chunk’s replicas.

  3. Task Tracker@each slave node

    A slave to Job Tracker. Lauches child processes(JVMs) to execute the map or reduce tasks.

  4. Data Node@each slave node

    A slave to the Name Node. Stores data on the local file system.

In Hadoop 2.x, the JobTracker and TaskTracker no longer exist and have been replaced by 3 components: (1) Resource Manager, (2) NodeManager, (3) ApplicationMaster.

Application Master(Per application): A supervisory task that requests resources needed for executing the tasks based on “containers”, responsible for tracking task status and monitoring for progress.

NodeManager: a worker daemon that can launch ApplicationMaster and task Containers.

Hadoop 1.x vs. Hadoop 2.x

Hadoop 1.x is slot-based task allocation, the resource of each slot is static. There are certain number of map slots and reduce slots.

Hadoop 2.x is container-based allocation, the resource(CPU and memeory) of each container is dynamic.

2. Hadoop Distributed File System(HDFS)

2. 1 HDFS design purpose
  1. Unreliable of single node: Hardware failure is the norm rather than execption.
  2. Files are huge: 100TB to PB datasets, multi-GB file sizes are the norm.
  3. File access model: read/append only, No random writes, only append to end.

Therefore, want to design a new file system exploit these new characteristics.

2.2 HDFS properties

  • Scale to tens of petabytes of storage
  • Files split into blocks(default 64 or 128MB), replica across several data nodes(default 3) for fault tolerance.
  • Single name node stores metadata(file names, block locations, etc.)
  • Optimized for large files, sequential reads
  • Emphasis is on high throughput of data access rather than low latency of data access.
